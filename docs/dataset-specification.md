# Спецификация датасета FishingLLM

## 1. Цель

Обучить модель рассуждать о рыбалке в Тверской области: анализировать факторы, обобщать опыт, взвешивать противоречия, строить рекомендации на основе данных, вести живой диалог.

## 2. Требования к модели (целевое поведение)

Модель должна уметь:

- Анализировать любые факторы (водоём, погода, сезон, давление, фаза луны, прозрачность воды, уровень воды, ветер, температура воды/воздуха, время суток, тип дна, течение) и делать выводы
- Рассуждать на основе фактов, а не просто отвечать на вопросы: «дано X, Y, Z → вывод»
- Обобщать данные из множества источников (отчёты, форумы, книги, статьи)
- Взвешивать противоречивые данные, указывать на неоднозначности
- Строить планы на рыбалку на основе набора вводных
- Признавать незнание: «у меня нет данных по этому водоёму», «есть только неподтверждённые сообщения»
- Запрашивать уточнения при недостатке информации
- Вести живой многообменный диалог (10+ реплик)
- Генерировать структурированные ответы (JSON для агентского режима)
- Быть вежливой, экспертной, но доступной

## 3. Ключевой принцип: от водоёма — к условиям — к рыбе — к рекомендации

Данные организуются иерархически, от общего к частному:

```
ВОДОЁМ
  └── Условия (сезон, погода, время, давление и т.д.)
       └── Целевая рыба
            └── Снасть / тактика
                 └── Результат / рекомендация
```

Пример:

```
Верхняя Волга, участок у д. Хотино
  ├── Июнь, t воды 18°C, уровень средний, течение слабое
  │   ├── Судак → джиг 12г, ступенчатая проводка, глубина 5м → улов 3 шт
  │   └── Лещ → фидер, кормушка 50г, червь → улов 2 кг
  ├── Июль, t воды 22°C, уровень низкий, течение слабое
  │   └── Щука → спиннинг, блесна, утро → улов 1 шт
  └── Сентябрь, t воды 12°C, уровень высокий, течение сильное
      └── Судак → джиг 18г, глубокие ямы → нет клёва
```

Модель должна учиться именно такому анализу: имея набор факторов, предсказывать наилучшую стратегию, а получив результат — корректировать будущие рекомендации.

## 4. Факторы (полный, открытый список)

Любой фактор, влияющий на рыбалку, должен учитываться.

**Обязательные** (должны быть в каждой записи analysis): водоём (название), сезон, целевая рыба.

**Опциональные** (чем больше, тем качественнее анализ):

- **Водоём (детали):** название, тип (озеро/река/вдхр), площадь, глубина, тип дна, прозрачность, уровень воды, течение, зарастаемость
- **Сезон:** весна (преднерест/нерест/посленерест), лето (начало/середина/конец), осень (начало/глубокая), зима (первый лёд/глухозимье/последний лёд)
- **Погода:** t воздуха, t воды, атмосферное давление (растёт/падает/стабильно), ветер (направление, сила), осадки, облачность
- **Время суток:** рассвет, утро, день, вечер, закат, ночь
- **Фаза луны:** новолуние, 1 четверть, полнолуние, последняя четверть
- **Гидрология:** паводок, межень, сброс воды ГЭС, цветение воды
- **Снасть:** спиннинг, фидер, поплавок, донка, мормышка, кружки, жерлицы
- **Приманка:** блесна, воблер, джиг, силикон, червь, опарыш, кукуруза, мотыль, мормыш
- **Тактика:** проводка (равномерная, ступенчатая, твичинг), точка ловли, прикормка
- **Рыба:** вид, размер, активность, фаза жизненного цикла

Список открытый — любые дополнительные факторы включаются по мере обнаружения.

## 5. Источники данных

Любые источники, содержащие релевантную информацию о рыбалке в Тверской области:

- Форумы и чаты (РусФишинг, Fishers.spb, VK, Telegram)
- Статьи и блоги (профильные сайты, YouTube-расшифровки)
- Книги (Сабанеев, Кузнецов, Матвеев, региональные издания)
- Научные статьи (ихтиология, гидробиология, лимнология)
- Отчёты о рыбалке (личные дневники, fishbrain, social media)
- Нормативные документы (правила рыболовства, приказы Минсельхоза)
- Устные истории (расшифровки интервью с опытными рыбаками)
- Метеоданные + logbook (сопоставление погоды и уловов)

Список не закрывается — любой источник с релевантными знаниями включается.

## 6. Data Provenance (происхождение данных)

Каждая запись в корпусе содержит метаданные о происхождении:

```json
{
  "id": "hex_id_8symbols",
  "source": "forum.russianfishing",
  "source_url": "https://...",
  "date": "2024-06-15",
  "license": "public_domain|fair_use|author_permission",
  "author": "username|book_author",
  "processed_by": "script_name|human_annotator",
  "source_reliability": 0.0-1.0
}
```

### Назначение полей

- `id` — уникальный идентификатор (hex, 8 символов, префикс по типу: `obs00001`, `ana00001`, `unc00001`, `dia00001`, `fac00001`, `exp00001`)
- `source` — краткое имя источника
- `source_url` — прямая ссылка (если доступна)
- `date` — дата создания записи или дата события
- `license` — правовой статус данных
- `author` — автор исходного текста
- `processed_by` — кто создал/разметил запись
- `source_reliability` — оценка надёжности источника (0.0 = непроверенный форум, 1.0 = рецензируемая статья)

Provenance позволяет при обнаружении ошибки найти и исправить или удалить все записи из проблемного источника.

## 7. Требования к объёму

Минимальный целевой объём — 50 МБ чистого текста, но жёсткой границы нет. Если данных мало — модель будет слабой. Если много — лучше. Ориентир: от 50 МБ до 500 МБ+ на финальную версию.

Первая итерация: 5-10 МБ для проверки пайплайна обучения.

## 8. Структура знаний

### Типы записей

Каждая запись в датасете относится к одному из типов. Ключевой тип — **analysis** (рассуждение на основе данных), именно ему учим модель в первую очередь.

#### fact — проверяемый факт

```json
{
  "type": "fact",
  "factor": {"water_body": "Селигер", "fish": "судак"},
  "content": "Судак на Селигере предпочитает глубины 4-8 м с каменистым дном.",
  "source": "Научный отчёт, 2021"
}
```

#### experience — эмпирическое наблюдение

```json
{
  "type": "experience",
  "factor": {"water_body": "Верхняя Волга", "season": "июнь", "tackle": "спиннинг"},
  "content": "На Верхней Волге в июне щука хорошо берёт на блесну утром.",
  "source": "Форум, пользователь Ivan23"
}
```

#### observation — первичное наблюдение (сырые данные)

```json
{
  "id": "obs00001",
  "type": "observation",
  "evidence": [
    {"factor": "water_body", "value": "Верхняя Волга, д. Хотино"},
    {"factor": "season", "value": "июнь"},
    {"factor": "water_temp", "value": "18°C"},
    {"factor": "tackle", "value": "джиг 12г"},
    {"factor": "result", "value": "2 судака, 1 окунь"}
  ],
  "source_url": "https://forum.example.com/report/123",
  "source_reliability": 0.6,
  "date": "2024-06-15"
}
```

Observation — это «что произошло» без интерпретации. Отделение первичных данных от анализа — ключевой принцип.

#### analysis — рассуждение на основе evidence (главный тип)

```json
{
  "id": "analysis00001",
  "type": "analysis",
  "evidence": [
    {"factor": "water_body", "value": "Верхняя Волга, участок у д. Хотино"},
    {"factor": "season", "value": "июнь"},
    {"factor": "water_temp", "value": "18°C"},
    {"factor": "depth", "value": "5м"},
    {"factor": "bottom", "value": "каменистое дно"},
    {"factor": "current", "value": "слабое"},
    {"factor": "pressure", "value": "стабильное"}
  ],
  "analysis": "1. Судак предпочитает глубокие ямы с каменистым дном — условие выполнено.\n2. В июне при 18°C судак активен, кормится у дна.\n3. По 15 отчётам с этого участка в аналогичных условиях судак стабильно брал на джиг 12-15г.\n4. Давление стабильное — клёв должен быть хорошим.",
  "recommendation": "Рекомендую ловлю на джиг 12-15г, ступенчатая проводка у дна, глубина 5-6м.",
  "source_reliability": 0.8,
  "analysis_confidence": 0.8,
  "source": "Анализ 15 отчётов с форума",
  "date": "2024-07-01"
}
```

Структура `evidence → analysis → recommendation` позволяет модели учиться видеть связь между фактами, рассуждением и выводом.

#### uncertainty — сомнение / противоречие

```json
{
  "id": "unc00001",
  "type": "uncertainty",
  "evidence": [
    {"factor": "water_body", "value": "Селигер"},
    {"factor": "season", "value": "июль"},
    {"factor": "fish", "value": "лещ"}
  ],
  "analysis": "1. По одним отчётам — лещ на Селигере в июле берёт на червя.\n2. По другим — на кукурузу.\n3. Нет достаточных данных, чтобы определить, от чего зависит выбор.\n4. Нужны дополнительные замеры.",
  "recommendation": "Точной рекомендации дать не могу. Червя и кукурузу стоит попробовать оба. Если расскажете подробнее об участке — смогу уточнить.",
  "source_reliability": 0.4,
  "analysis_confidence": 0.3,
  "source": "Сводка по форумам, 5 источников"
}
```

#### dialogue — диалог (ChatML)

```json
{
  "type": "dialogue",
  "messages": [
    {"role": "user", "content": "Планирую рыбалку на Селигере в июне. Хочу поймать судака. Что посоветуешь?"},
    {"role": "assistant", "content": "Отличный выбор! На Селигере в июне судак активно кормится после нереста. Какими снастями располагаете и с лодки или с берега планируете ловить?"},
    {"role": "user", "content": "С лодки, спиннинг есть, джиговый."},
    {"role": "assistant", "content": "Прекрасно. На Селигере судак держится на глубине 4-8 м у каменистых гряд. Рекомендую участки у островов — например, у о. Хотино. Джиг 12-15 г, ступенчатая проводка. Лучшее время — раннее утро или перед закатом. По прогнозу давление стабильное — клёв должен быть хорошим."},
    {"role": "user", "content": "А на что ещё можно рассчитывать на этом участке?"},
    {"role": "assistant", "content": "На том же участке можно поймать щуку — она держится у границы камыша. Если поставить поводок, можно попробовать воблер. Также есть шанс на крупного окуня. Но если цель именно судак — лучше чисто джиговая оснастка без поводка."}
  ]
}
```

## 9. Reasoning не привязан к вопросу

Модель должна учиться рассуждать **от фактов к выводу**, а не от вопроса к ответу. Разница принципиальная:

❌ **Вопрос-ответ:**  
«Где ловить судака?» → «На глубине 4-8 м».  
Модель просто запомнила паттерн.

✅ **Рассуждение от данных:**  
Дано: водоём X, глубина Y, дно Z, сезон W, t воды T, давление P.  
→ Судак предпочитает...  
→ В данных условиях...  
→ По отчётам...  
→ Вывод: рекомендую...

Поэтому **analysis** — главный тип записей. Диалоги и вопросы-ответы — дополнительные форматы, но ядро датасета — пары «набор факторов → рассуждение → вывод».

## 10. Диалоги

Минимум 3-5 обменов — это нижняя граница. Реальные диалоги могут быть на 10-20+ реплик. Модель должна уметь:

- Поддерживать контекст на протяжении длительного разговора
- Возвращаться к ранее сказанному
- Менять тему плавно
- Запоминать предпочтения пользователя в рамках диалога

Пример длинного диалога:

```
User: Планирую поездку на Селигер в июне.
Ass: Отлично! С какой целью? Просто отдохнуть или целенаправленно порыбачить?
User: Порыбачить. Хочу судака попробовать поймать.
Ass: Судак на Селигере — отличная цель. У вас есть лодка?
User: Да, своя, моторная.
Ass: Ещё лучше. Тогда рекомендую участки у островов — там глубины 4-8 м...
User: А какие приманки лучше?
Ass: Зависит от времени суток. Утром и вечером — джиг 12-15г...
User: А если погода испортится?
Ass: При ухудшении погоды судак уходит на глубину до 10-12 м...
...и так далее (10-15 реплик)
```

## 11. Фильтрация и очистка



1. **Unicode нормализация** — NFC
2. **Удаление HTML/разметки**
3. **Фильтрация токсичности** — удалить оскорбления и перепалки (ad hominem, угрозы). Нейтральные цитаты и отчёты с отдельными грубыми словами не удалять, чтобы не искажать материал
4. **Дедупликация** — exact match (первая итерация). MinHash/SimHash — по мере роста корпуса
5. **Удаление слишком коротких записей** — < 100 символов (кроме диалогов)
6. **Lang-filter** — только русский язык

### Целевая структура по типам (ориентир, не жёсткое требование)

| Тип | Доля | Назначение |
|-----|------|-----------|
| analysis | ~40% | Рассуждение на основе факторов — ядро обучения |
| observation | ~15% | Сырые наблюдения, первичные данные |
| dialogue | ~20% | Диалоги в ChatML-формате |
| fact | ~10% | Проверяемые факты |
| experience | ~10% | Эмпирические наблюдения |
| uncertainty | ~5% | Сомнения и противоречия

## 12. Структура на диске

```
data/
  raw/
    forums/
    books/
    articles/
    reports/
    interviews/
  processed/
    corpus.jsonl     # все записи
    factors.json     # таксономия факторов и их значений
  splits/
    train.jsonl      # 80%
    valid.jsonl      # 10%
    test.jsonl       # 10%
```

## 13. Train/Validation/Test split

Разбиение выполняется **по источникам/документам**, а не случайно по строкам. Это предотвращает утечку: почти одинаковые записи из одного источника не попадут одновременно в train и validation.

Правило: все записи из одного источника (одна тема форума, одна статья, одна книга) целиком попадают в один split.

## 14. Инструменты

| Задача | Инструмент |
|--------|-----------|
| Очистка HTML | BeautifulSoup / selectolax |
| Детекция языка | langdetect / fastText |
| Дедупликация (первая итерация) | exact match по содержимому |
| Дедупликация (масштабирование) | datasketch (MinHash) |
| Фильтрация токсичности | регулярки + список стоп-слов |
| Разметка | скрипты + ручная разметка для сложных случаев |
| Хранение | JSONL (одна запись = одна строка JSON) |

## 15. Первая итерация

1. Собрать 1-2 источника (один форум, одну статью)
2. Очистить → получить 0.5-1 МБ сырого текста
3. Вручную подготовить и тщательно проверить первые 10-20 записей типа observation (сырые данные из источников)
4. Вручную подготовить первые 10-20 записей типа analysis (evidence → analysis → recommendation) — они станут эталоном для автоматизации
5. Вручную подготовить 5-10 коротких диалогов (3-6 реплик)
6. Проверить пайплайн: сбор → очистка → токенизация (FishingTokenizer) → обучение пробной модели
7. Оценить качество → итерация

**Важно на первой итерации:** качество важнее объёма. 20 эталонных записей ценнее 200 автоматически сгенерированных с ошибками.

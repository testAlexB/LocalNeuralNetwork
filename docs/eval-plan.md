# План оценки качества FishingLLM v2

## 1. Цель

Измерять прогресс модели по всем измерениям spec v1.3: факты, каузальное мышление, синтез свидетельств, формальные гипотезы, машинно-исполнимые рекомендации, диагностика провалов, калибровка уверенности, граф-консистентность.

## 2. Уровни оценки

```
Уровень 1: Во время обучения
    ↓
Уровень 2: Автоматические тесты после чекпоинта
    ├── Held-out (категоризированный)
    ├── Bенчмарки (8 категорий)
    ├── Граф-консистентность
    └── Калибровка уверенности
    ↓
Уровень 3: Экспертная оценка (человек)
    ↓
Уровень 4: Мониторинг в инференсе
```

## 3. Уровень 1 — Во время обучения

### train/val loss

Стандартный контроль early stopping.

### Perplexity (PPL)

`PPL = exp(cross_entropy_loss)`. Для 150M на узком домене:
- PPL < 20 → отлично
- PPL 20-50 → приемлемо
- PPL > 50 → проблемы

### Token accuracy

Процент угаданных токенов.

### loss по типам записей

Контроль отдельно по:
- analysis (особенно rejected)
- evidence_synthesis
- hypothesis (formal_rule)
- outcome (execution_match, failure_reason)

Если loss на rejected analysis растёт — модель перекашивается в подтверждение гипотез.

## 4. Уровень 2 — Автоматические тесты

### 4.1 Held-out набор (80-120 вопросов)

| Категория | Пример | Критерий |
|-----------|--------|----------|
| Факт | «Когда нерестится щука?» | Точность (месяц, t) |
| Рекомендация | «Снасть для судака в июне?» | Обоснованность, учёт факторов |
| Сомнение | «Какая наживка для леща?» | Признание неоднозначности |
| Незнание | «Сколько щук поймали в 2023?» | Признание незнания |
| Уточнение | «Где ловить?» | Запрос деталей |
| Вежливость | Грубый тон | Вежливый ответ |
| Каузальность | «После дождя клевало — дождь причина?» | Различие корреляции и причинности |
| Гипотеза | «Почему судак не клюёт?» | Формальная гипотеза (variable, operator, value) |
| Отклонение | «Полнолуние — лучшее время?» | Отклонение при отсутствии данных |
| Синтез | «Одни говорят — утром, другие — вечером» | Взвешивание свидетельств |
| Альтернатива | «Температура — главный фактор?» | Упоминание альтернатив |
| Диагностика | «Был по рекомендации — не поймал» | Анализ failure_reason |
| Исполнимость | «Что делать?» | JSON с actions[] |

### 4.2 Метрики для каждого ответа

1. **Factual accuracy** — Exact match ключевых фактов
2. **Uncertainty detection** — Признание незнания в нужных случаях
3. **Toxicity** — 0% мата
4. **Politeness** — Наличие вежливых форм
5. **Repetition** — Доля повторяющихся n-грамм
6. **JSON validity** — Валидный JSON при запросе
7. **Causal correctness** — Не путает корреляцию с причинностью (probe-questions)
8. **Hypothesis formality** — Содержит ли гипотеза variable + operator + value
9. **Alternative count** — Число альтернативных объяснений (≥1 где уместно)
10. **Action executability** — actions[].type ∈ допустимых, value согласован с текстом
11. **Failure reason validity** — failure_reason ∈ словаря failure_reasons.json

### 4.3 Бенчмарки (8 категорий)

Каждый бенчмарк — JSONL с prompt + expected + метрикой.

| Бенчмарк | Что проверяет | Метрика |
|----------|--------------|---------|
| `reasoning` | Многошаговое рассуждение от данных к выводу | Правильность цепочки + каузальных связей |
| `planning` | Построение плана рыбалки по вводным | Полнота (depth, tackle, time, conditions) |
| `memory` | Удержание контекста в диалоге 10+ реплик | Факты не противоречат ранним репликам |
| `retrieval` | Поиск релевантных фактов по запросу | Precision/Recall фактов |
| `dialogue` | Естественность диалога, вежливость | Экспертная оценка (1-5) |
| `hallucination` | Отказ от ответа при отсутствии данных | false positive rate |
| `uncertainty` | Корректная калибровка уверенности | ECE (Expected Calibration Error) |
| `causal_reasoning` | Различение корреляции и причинности | Accuracy на probe-вопросах |

**Пример probe для causal_reasoning:**
```json
{
  "prompt": "После дождя всегда хорошо клюёт. Значит ли это, что дождь — причина клёва?",
  "expected_hint": "корреляция",
  "expected_reject": true,
  "metric": "causal_correctness"
}
```

### 4.4 Граф-консистентность

Автоматическая проверка:
- Каждый `based_on` имеет соответствующую `relation` с правильным типом
- relation.from → relation.to — существующие id
- relation.relation ∈ {supports, contradicts, derived_from, generated_from, validated_by, background}
- Нет циклов в графе derived_from

### 4.5 Калибровка уверенности

**ECE (Expected Calibration Error)** — мера того, насколько analysis_confidence совпадает с фактической точностью.

Разбиваем предсказания на 10 бинов по confidence (0-0.1, 0.1-0.2, ..., 0.9-1.0). Для каждого бина:

```
ECE = Σ |accuracy(bin) - avg_confidence(bin)| * weight(bin)
```

Цель: ECE < 0.1.

Дополнительно: reliability diagram (график accuracy vs confidence).

## 5. Уровень 3 — Экспертная оценка

### 5.1 Шкала

Для каждого диалога (10-20 на чекпоинт):

| Критерий | Вес | 0 плохо | 1 средне | 2 хорошо |
|----------|-----|---------|----------|----------|
| Корректность фактов | 0.20 | Ошибки | Неточности | Всё верно |
| Аргументация | 0.15 | Нет рассуждения | Поверхностно | Каузальные связи + альтернативы |
| Признание незнания | 0.10 | Ложь | Уклончиво | «Не знаю» + уточнение |
| Каузальная корректность | 0.15 | correlation=causation | Не уверен | Чёткое различение |
| Синтез свидетельств | 0.10 | Игнорирует противоречия | Упомянул | Взвесил с weights |
| Исполнимость | 0.10 | Нет действий | Есть текст | Есть actions[] |
| Диагностика провалов | 0.05 | Не анализирует | Поверхностно | failure_reason + execution_match |
| Вежливость | 0.05 | Грубость | Сухо | Вежливо |
| Полезность | 0.10 | Бесполезно | Частично | Конкретная рекомендация |

Итог: weighted sum (0-2).

### 5.2 Экспертная панель

- Пользователь — первая линия
- Рыбаки/эксперты — по возможности
- Сильная модель как прокси-эксперт: «оцени каузальную корректность 1-5»

## 6. Уровень 4 — Мониторинг в инференсе

### 6.1 Логирование

```json
{
  "id": "req_00001",
  "prompt": "...",
  "response": "...",
  "timestamp": "...",
  "temperature": 0.7,
  "top_p": 0.9,
  "latency_ms": 1234,
  "repetition_score": 0.05,
  "toxicity_score": 0.001,
  "has_actions": true,
  "actions_valid": true,
  "confidence_avg": 0.72,
  "causal_links_count": 2,
  "alternative_explanations_count": 1
}
```

### 6.2 Триггеры

- toxicity > 0.1 → ручная проверка
- repetition > 0.3 → снизить temperature
- actions_valid = false → проверить рекомендацию
- has_actions = false где ожидались → дообучение
- causal_links_count = 0 где ожидались → проверить reasoning
- `<unk>` → проверить токенизатор
- latency > 5s → оптимизация

### 6.3 Обратная связь

- 👍 полезно
- 👎 неверно
- 🤷 не по теме
- 🧠 каузальная ошибка (новый: для сбора примеров correlation≠causation)

## 7. План по итерациям

### Итерация 0 (первый запуск)

- Только val loss, PPL
- 5 ручных тестов
- Метрика: осмысленный текст, без зацикливания

### Итерация 1 (первый датасет)

- Held-out (факты, вежливость, токсичность)
- Бенчмарки: reasoning, dialogue
- 10 экспертных диалогов
- Цель: factual accuracy > 60%, toxicity = 0%, PPL < 50

### Итерация 2 (расширенный датасет + каузальность)

- Все 8 бенчмарков
- Калибровка (ECE)
- Граф-консистентность
- Гипотезы (formality)
- Цель: factual accuracy > 80%, causal correctness > 70%, ECE < 0.15

### Итерация 3 (финальная)

- Все метрики Уровня 2 + 3
- Мониторинг инференса
- Цель: causal correctness > 85%, ECE < 0.1, action validity > 90%
- Ready for agent mode

## 8. Автоматизация

- GitHub Actions: evaluate.py после каждого обучения
- Сравнение PPL и ECE между коммитами
- Регрессия: factual accuracy -5% → не мержить
- Регрессия: causal correctness -10% → не мержить
- Dashboard: метрики по чекпоинтам + reliability diagram

## 9. Что НЕ измеряем в первую очередь

- Время инференса (оптимизируем потом)
- Размер модели (фиксирован 150M)
- Скорость токенизации
- Разнообразие лексики

# План оценки качества FishingLLM

## 1. Цель

Измерять прогресс модели при обучении и гарантировать, что её ответы корректны, аргументированы, вежливы и полезны.

## 2. Уровни оценки

```
Уровень 1: Во время обучения
    ↓
Уровень 2: Автоматические тесты после чекпоинта
    ↓
Уровень 3: Экспертная оценка (человек)
    ↓
Уровень 4: Мониторинг в инференсе (RLHF-ready)
```

## 3. Уровень 1 — Во время обучения

### train/val loss

Стандартный контроль: если val loss перестаёт снижаться или начинает расти — early stopping.

### Perplexity (PPL)

`PPL = exp(cross_entropy_loss)` — чем ниже, тем увереннее модель.

Для модели 150M на узком домене:
- PPL < 20 → отлично
- PPL 20-50 → приемлемо
- PPL > 50 → данные или обучение проблемные

### Accuracy на предсказании следующего токена (token-level)

Мера: какой процент токенов модель угадала точно.

## 4. Уровень 2 — Автоматические тесты

### 4.1 Эталонный тестовый набор (held-out)

50-100 вопросов с эталонными ответами, разбитых по категориям:

| Категория | Пример вопроса | Критерий оценки |
|-----------|---------------|-----------------|
| Факт | «Когда нерестится щука в Тверской области?» | Точность (месяц, t воды) |
| Рекомендация | «Какую снасть выбрать для судака на Селигере в июне?» | Обоснованность, учёт факторов |
| Сомнение | «Какая наживка лучше для леща?» | Признание неоднозначности |
| Незнание | «Сколько щук поймали на Селигере в 2023 году?» | Признание незнания |
| Уточнение | «Где ловить рыбу?» | Запрос деталей |
| Вежливость | Любой вопрос с нецензурным/грубым тоном | Вежливый ответ |

### 4.2 Метрики для каждого ответа

Автоматически (через скрипт):

1. **Factual accuracy** — Exact match ключевых фактов (числа, названия, месяцы)
2. **Uncertainty detection** — Сказала ли модель «не знаю», «нет данных», «возможно» в нужных случаях
3. **Toxicity filter** — Есть ли мат, оскорбления в ответе (должно быть 0)
4. **Politeness** — Есть ли «здравствуйте», «пожалуйста», обращение по правилам русского языка
5. **Repetition** — Доля повторяющихся n-грамм (штраф за зацикливание)
6. **JSON validity** — Если запрошен JSON — валидный ли JSON

## 5. Уровень 3 — Экспертная оценка

### 5.1 Шкала экспертной оценки

Для каждого диалога (10-20 диалогов на чекпоинт):

| Критерий | Вес | 0 (плохо) | 1 (средне) | 2 (хорошо) |
|----------|-----|-----------|------------|------------|
| Корректность фактов | 0.3 | Ошибки в базовых фактах | Мелкие неточности | Всё верно |
| Аргументация | 0.25 | Нет рассуждения | Поверхностное | Глубокое, со взвешиванием |
| Признание незнания | 0.15 | Выдаёт ложь за факт | Уклончиво | Честно «не знаю» + уточнение |
| Вежливость | 0.1 | Грубость | Сухо | Вежливо, экспертно |
| Полезность | 0.2 | Бесполезный ответ | Частично полезен | Конкретная рекомендация |

Итоговая оценка: weighted sum (0-2 балла).

### 5.2 Экспертная панель

- Сам пользователь (ты) — первая линия
- Другие рыбаки/эксперты — по мере возможности
- ChatGPT как прокси-эксперт — быстрый фильтр: «оцени ответ модели на шкале 1-5»

## 6. Уровень 4 — Мониторинг в инференсе

### 6.1 Логирование

Каждый запрос к модели логируется:

```json
{
  "id": "req_00001",
  "prompt": "...",
  "response": "...",
  "timestamp": "...",
  "temperature": 0.7,
  "top_p": 0.9,
  "latency_ms": 1234,
  "repetition_score": 0.05,
  "toxicity_score": 0.001
}
```

### 6.2 Автоматические триггеры

- Если `toxicity_score > 0.1` → тегнуть на ручную проверку
- Если `repetition_score > 0.3` → снизить temperature для следующего запроса
- Если ответ содержит `<unk>` token → проверить токенизатор
- Если latency > 5s на CPU → оптимизировать

### 6.3 Обратная связь от пользователя

После каждого ответа — опциональная кнопка:

- 👍 полезно
- 👎 неверно
- 🤷 не по теме

Сбор статистики: соотношение 👍/👎 по категориям вопросов.

## 7. План оценки по итерациям

### Итерация 0 (первый пробный запуск)

- Только validation loss и PPL
- 5 ручных тестовых диалогов
- Метрика: «модель не зацикливается, даёт осмысленный текст»

### Итерация 1 (первый полноценный датасет)

- Автоматические тесты (факты, вежливость, токсичность)
- 10 экспертных диалогов
- Метрика: factual accuracy > 60%, toxicity = 0%

### Итерация 2 (расширенный датасет + reasoning)

- Полный held-out набор (50+ вопросов)
- Оценка аргументации экспертом
- Метрика: factual accuracy > 80%, uncertainty detection > 70%

### Итерация 3 (финальная)

- Все метрики Уровня 2 и 3
- Инференс-мониторинг
- Ready for agent mode

## 8. Автоматизация (будущее)

- GitHub Actions: после каждого обучения запускать `evaluate.py`
- Сравнение PPL между коммитами
- Регрессионный тест: если factual accuracy упала >5% — не мержить
- Dashbord с метриками по чекпоинтам

## 9. Что измеряем НЕ в первую очередь

- Время инференса (оптимизируем, когда модель работает)
- Размер модели (он фиксирован: 150M)
- Скорость токенизации (достаточно быстра)
- Разнообразие лексики (пока не принципиально)
